## 摘要

随着中小商贸企业数字化经营需求不断提升，销售数据已成为企业进行库存控制、促销安排和经营决策的重要依据。然而，传统人工统计和经验判断方式难以高效处理长期、多门店、多品类销售数据，也难以对未来销售趋势形成稳定、可解释的预测结果。针对上述问题，本文设计并实现了一套商贸公司销售数据分析与智能预测系统。系统采用 Python Flask 构建 Web 后端，结合 Pandas、NumPy 完成数据导入、校验、清洗、聚合和特征工程，使用 Bootstrap 与 ECharts 构建交互式分析页面，并集成 ARIMA、Prophet 和 LSTM 三类时间序列预测模型，实现销售分析、智能预测、模型评估、报告展示和 CSV 导出等功能。

本文以 Kaggle Favorita Store Sales 公开零售数据集为实验数据基础，选取 5 家门店、33 个商品品类、2013 年 1 月 1 日至 2017 年 8 月 15 日共 277860 条销售记录作为研究对象。系统围绕“数据导入-数据清洗-销售分析-预测配置-后台建模-报告展示-结果导出”的业务流程展开设计，后端提供 23 个页面与 API 路由，前端实现数据上传、运营看板、预测工作台和决策报告四个主要页面。预测模块通过 ADF 检验、ARIMA 自动定阶、Prophet 节假日和促销回归量建模、LSTM 滑动窗口训练以及 MAE、RMSE、MAPE、SMAPE、R² 等指标对比，形成多模型预测与评估闭环。

系统测试采用 pytest 完成数据处理、模型模块和 API 集成验证，覆盖文件上传、字段校验、异常数据处理、模型预测、模型缓存、任务取消和结果导出等场景。最新测试结果为 44 个测试用例全部通过，表明系统主要功能流程具备较好的正确性、稳定性和可扩展性。研究结果说明，将 Web 可视化系统与多模型时间序列预测方法结合，能够为中小商贸企业提供一种低成本、易部署、可解释的销售数据分析与预测工具，对提升销售决策效率具有一定实践价值。

## Abstract

With the continuous development of digital operation in small and medium-sized trading enterprises, sales data has become an important basis for inventory control, promotion planning, and business decision-making. However, traditional manual statistics and experience-based judgment are inefficient when dealing with long-term sales records involving multiple stores and product families, and they can hardly provide stable and interpretable forecasts for future sales trends. To address these problems, this thesis designs and implements a sales data analysis and intelligent forecasting system for trading companies. The system uses Python Flask as the Web backend, Pandas and NumPy for data importing, validation, cleaning, aggregation, and feature engineering, and Bootstrap with ECharts for interactive visualization. It integrates three time series forecasting models, namely ARIMA, Prophet, and LSTM, and provides sales analysis, intelligent forecasting, model evaluation, report presentation, and CSV export functions.

The experimental data is based on the public Kaggle Favorita Store Sales dataset. A subset containing 277,860 sales records from 5 stores and 33 product families between January 1, 2013 and August 15, 2017 is used in this project. The system is designed around the business workflow of data importing, data cleaning, sales analysis, forecast configuration, background modeling, report presentation, and result export. The backend provides 23 page and API routes, while the frontend contains four main pages: data importing, operation dashboard, forecasting workspace, and decision report. The forecasting module forms a complete multi-model evaluation loop through ADF testing, automatic ARIMA order selection, Prophet holiday and promotion modeling, LSTM sliding-window training, and comparison metrics including MAE, RMSE, MAPE, SMAPE, and R-squared.

The system is tested with pytest, covering data processing, model modules, and API integration scenarios, including file uploading, field validation, abnormal data handling, model forecasting, model caching, task cancellation, and result exporting. The latest verification result shows that all 44 test cases passed, indicating that the main functional workflow has good correctness, stability, and extensibility. The results demonstrate that combining a Web visualization system with multiple time series forecasting models can provide small and medium-sized trading enterprises with a low-cost, easy-to-deploy, and interpretable sales analysis and forecasting tool, which has practical value for improving sales decision efficiency.

**Keywords**: sales data analysis; time series forecasting; Flask; ARIMA; Prophet; LSTM; ECharts

## 第 1 章 绪论

### 1.1 研究背景

近年来，随着信息技术、电子商务和数字化经营模式的快速发展，商贸企业在采购、销售、库存、促销和客户服务等环节持续产生大量业务数据。对于中小商贸企业而言，销售数据是企业经营管理中最直接、最重要的数据资产之一。通过对历史销售数据进行分析，企业可以了解不同商品品类、不同门店和不同时间周期下的销售变化规律，为备货计划、促销安排、库存控制和经营策略制定提供依据。

然而，在实际经营过程中，许多中小商贸企业仍然依赖人工经验和简单报表进行销售判断。一方面，传统表格工具在处理长期、多品类、多门店销售数据时效率较低，难以直观展示复杂的趋势、周期性和结构变化；另一方面，仅依靠经验进行销售预测容易受到主观因素影响，难以及时发现销售波动背后的规律，导致库存积压、缺货、促销效果不佳等经营风险。

随着时间序列分析、机器学习和 Web 可视化技术的发展，将销售数据分析、智能预测和可视化展示集成到统一系统中已成为提升企业数据化经营能力的重要方向。ARIMA、Prophet、LSTM 等模型分别从统计建模、趋势季节性分解和深度学习序列建模角度提供了不同的预测思路。结合 Flask、Pandas、ECharts 等技术，可以构建轻量化、易部署、交互友好的销售预测系统，使企业管理者能够通过浏览器完成数据导入、分析、预测和结果查看。

基于上述背景，本文设计并实现一套面向中小商贸企业的销售数据分析与智能预测系统，旨在将数据处理、统计分析、模型预测和可视化展示结合起来，为企业提供一体化的数据分析与决策辅助工具。

### 1.2 研究意义

本课题的研究意义主要体现在以下几个方面。

第一，有助于提升中小商贸企业的数据利用效率。系统能够将原始销售记录转化为趋势图、品类占比图、门店销售统计、促销效果分析等可视化结果，降低业务人员理解数据的难度。

第二，有助于提升销售预测的科学性。系统集成 ARIMA、Prophet 和 LSTM 等预测模型，通过统一指标对模型进行对比评价，避免单一模型或人工经验带来的局限性。

第三，有助于降低企业经营风险。通过对未来销售趋势进行预测，企业可以提前安排库存和促销计划，减少库存积压和缺货风险，提高资源配置效率。

第四，具有一定工程实践价值。系统采用 Flask Web 架构，结合前后端交互、后台任务管理、模型缓存、图表展示和数据导出等功能，体现了数据分析系统从算法到应用落地的完整开发过程。

### 1.3 国内外研究现状

在销售预测领域，传统统计模型与机器学习模型均得到了广泛应用。ARIMA 模型是经典的时间序列预测方法，能够通过自回归、差分和移动平均项对平稳时间序列进行建模，在销售量、库存量和经济指标预测中具有较强的解释性。其优点是理论成熟、参数含义明确、适用于中短期线性趋势预测；不足在于对非线性关系、复杂季节性和外部影响因素的表达能力有限。

Prophet 模型由 Facebook 开源，适用于具有趋势、周期性、节假日效应的业务时间序列。该模型将时间序列分解为趋势项、季节项、节假日项和误差项，具有参数较少、易调节、对缺失值和异常点相对鲁棒等特点。对于零售销售这类受节假日和周期性影响较明显的数据，Prophet 具有较好的应用价值。

随着深度学习的发展，LSTM 等循环神经网络模型被广泛用于序列预测任务。LSTM 通过门控机制缓解传统循环神经网络中的长期依赖问题，能够学习复杂的非线性时序特征。在销售预测场景中，LSTM 可用于捕捉长期趋势和局部波动，但其训练成本较高，对样本规模、参数设置和计算资源有一定要求。

从系统实现角度看，已有不少研究围绕 Python 数据分析平台、销售预测系统和可视化报表工具展开。但部分系统只关注模型算法，缺少完整的 Web 交互和可视化展示；部分系统只提供数据分析报表，缺少多模型预测和评价能力。本文课题在开题报告既定技术路线基础上，将数据清洗、销售分析、多模型预测、模型对比和 Web 可视化整合到统一系统中，形成较完整的应用闭环。

### 1.4 研究内容

本文主要研究内容如下。

（1）销售数据预处理方法研究。针对 Favorita 销售数据中的日期、门店、品类、销量和促销字段，完成数据加载、字段校验、编码兼容、重复记录聚合、缺失值处理、异常值处理和时间序列切分等处理流程。

（2）销售数据可视化分析方法研究。围绕销售总量、日均销量、同比增长、品类占比、Top 品类排行、品类相关性、星期效应、促销效果等指标设计分析接口和图表展示方式。

（3）多模型销售预测方法研究。实现 ARIMA、Prophet 和 LSTM 三种预测模型，分别利用统计模型、趋势季节模型和深度学习模型完成单品类单门店销售预测。

（4）模型评价与预测报告研究。设计 MAE、RMSE、MAPE、SMAPE、R² 和综合评分等评价指标，对不同模型预测结果进行对比，生成预测曲线、误差表格、雷达图和推荐说明。

（5）Web 系统设计与实现。采用 Flask 构建后端接口，使用 Bootstrap 和 ECharts 实现前端页面，完成数据上传、分析看板、预测工作台和预测报告等功能模块。

## 第 2 章 相关技术

### 2.1 Flask Web 框架

Flask 是 Python 语言中常用的轻量级 Web 框架，具有结构简单、扩展灵活、学习成本低等特点。Flask 通过路由函数处理 HTTP 请求，可以方便地构建 Web 页面和 REST 风格接口。对于毕业设计中的数据分析系统而言，Flask 能够承担前端页面渲染、文件上传、API 服务、预测任务启动和结果返回等功能。

本系统使用 Flask 构建后端应用，主要提供首页、分析页、预测页和报告页等页面路由，同时提供数据上传、销售分析、预测任务、预测结果和导出文件等 API 接口。系统将耗时的模型训练和预测过程放入后台线程中执行，并通过任务管理模块维护预测进度，使前端能够轮询任务状态并实时展示执行进度。

### 2.2 Pandas 与 NumPy

Pandas 是 Python 数据分析领域常用的数据处理库，提供 DataFrame 和 Series 等数据结构，适合进行结构化数据读取、清洗、分组聚合和时间序列处理。NumPy 提供高效的数组计算能力，是 Pandas、机器学习和深度学习模型计算的重要基础。

本系统使用 Pandas 读取 CSV 和 XLSX 文件，对销售数据进行字段校验、日期解析、数值转换、重复行聚合、按品类和门店分组处理、按时间粒度聚合等操作。NumPy 主要用于模型输入数组构造、评价指标计算、预测结果裁剪和数值安全处理。

### 2.3 ECharts 可视化技术

ECharts 是常用的前端图表库，支持折线图、柱状图、饼图、热力图、雷达图等多种图表类型，适合构建交互式数据可视化页面。本系统前端通过 ECharts 展示销售趋势、月度对比、品类占比、Top 品类排行、预测曲线、模型误差对比和雷达图等内容。通过图表化展示，用户能够更直观地理解销售数据变化规律和模型预测结果。

### 2.4 ARIMA 模型

ARIMA 模型全称为自回归积分滑动平均模型，由 AR、I、MA 三部分组成。其中 AR 表示自回归项，用于描述当前值与过去值之间的关系；I 表示差分项，用于将非平稳序列转换为平稳序列；MA 表示移动平均项，用于描述误差项之间的关系。ARIMA 通常表示为 ARIMA(p,d,q)，其中 p 为自回归阶数，d 为差分阶数，q 为移动平均阶数。

本系统使用 ADF 检验判断序列平稳性，并根据 AIC 或 BIC 准则在给定范围内搜索较优的 p 和 q 参数。ARIMA 模型具有解释性强、训练速度相对较快等优点，适合作为销售预测系统中的基础统计模型。

### 2.5 Prophet 模型

Prophet 是一种面向业务时间序列预测的模型，其基本思想是将时间序列分解为趋势项、季节项、节假日项和误差项。Prophet 能够处理非线性增长趋势、周期性变化、缺失值和异常点，并支持加入节假日和外部回归变量。

Favorita 销售数据中包含节假日文件和促销字段，因此本系统在 Prophet 模块中设计了节假日加载逻辑，并支持将 onpromotion 作为可选回归变量参与建模。Prophet 模型输出除预测值外，还可以提供趋势、周季节性、年季节性和节假日分量，为用户理解预测结果提供参考。

### 2.6 LSTM 模型

LSTM 是长短期记忆网络，是循环神经网络的一种改进结构。LSTM 通过输入门、遗忘门和输出门控制信息在时间序列中的传递，能够缓解普通循环神经网络难以学习长期依赖的问题。

本系统使用 PyTorch 构建 LSTM 网络，将销售序列转换为滑动窗口样本进行训练。模型采用 MinMaxScaler 对训练序列归一化，使用 Adam 优化器训练，并引入学习率调整和早停机制。LSTM 模型作为扩展对比模块，可用于探索非线性销售序列建模效果。

### 2.7 模型评价指标

为统一比较不同模型预测效果，系统使用以下评价指标。

（1）MAE，平均绝对误差，反映预测值与真实值之间的平均绝对偏差。

（2）RMSE，均方根误差，对较大误差更加敏感。

（3）MAPE，平均绝对百分比误差，便于从百分比角度评价预测偏差。系统在实际值为 0 时跳过该点，以避免除零问题。

（4）SMAPE，对称平均绝对百分比误差，可作为含零序列场景下的补充指标。

（5）R²，决定系数，用于衡量模型对真实序列波动的解释程度。

系统在上述指标基础上计算综合评分，并根据综合评分对模型进行排序和推荐。

## 第 3 章 系统需求分析

### 3.1 系统用户分析

本系统面向具有销售数据分析需求的中小商贸企业管理人员、运营人员和数据分析人员。用户通常希望通过简单的 Web 界面上传销售数据，快速查看销售概况、趋势变化和品类结构，并通过预测模型获取未来一段时间的销售趋势参考。因此系统需要兼顾操作简便性、分析结果直观性和预测结果可解释性。

### 3.2 功能需求分析

结合开题报告目标和项目实现情况，系统主要功能需求如下。

（1）数据导入功能。用户可以上传 CSV 或 XLSX 格式销售数据，系统对文件格式、字段完整性、编码和数据内容进行校验，并给出错误或警告提示。

（2）数据清洗功能。系统需要对日期字段进行解析，对销售量和促销字段进行数值转换，对重复记录进行聚合，对负值进行修正，对缺失值和异常值进行处理。

（3）数据分析功能。系统应提供总销量、日均销量、同比增长、最佳品类、销售趋势、月度对比、品类销售占比、Top 品类排行、相关性、星期效应和促销效果等分析能力。

（4）预测配置功能。用户可以选择预测品类、门店、预测天数和预测模型，并设置 ARIMA、Prophet、LSTM 的部分参数。

（5）模型预测功能。系统应支持 ARIMA、Prophet 和 LSTM 模型训练与预测，并能够处理模型不可用、参数异常、序列过短等情况。

（6）任务进度功能。模型训练可能耗时较长，系统需要提供后台任务状态，包括当前步骤、进度百分比和各模型执行状态。

（7）预测报告功能。系统应展示未来预测曲线、误差指标表格、模型误差柱状图、雷达图、Prophet 分量图、LSTM 损失曲线和推荐模型说明。

（8）结果导出功能。系统应支持将预测结果导出为 CSV 文件，便于后续保存和分析。

### 3.3 非功能需求分析

（1）易用性。系统界面应简洁直观，用户能够按照“上传数据—查看分析—配置预测—查看报告”的流程完成操作。

（2）可靠性。系统应能处理缺列、负值、重复行、空会话、非法模型、预测天数越界等异常情况，并返回明确提示。

（3）可扩展性。系统模型模块相对独立，后续可扩展 XGBoost、随机森林、Transformer 等模型，也可扩展批量预测和自动报表功能。

（4）可解释性。系统不仅提供预测值，还提供模型误差、模型排序、Prophet 分量和指标对比，帮助用户理解模型效果。

（5）离线可用性。系统前端使用本地 Bootstrap 和 ECharts 静态资源，能够减少对外部 CDN 的依赖。

### 3.4 数据需求分析

本系统使用 Kaggle Favorita Store Sales 数据集的 5 家门店子集，核心字段包括 date、store_nbr、family、sales 和 onpromotion。其中 date 表示日期，store_nbr 表示门店编号，family 表示商品品类，sales 表示销售量，onpromotion 表示促销数量。系统还使用 holidays_events.csv 识别节假日和事件信息，用于异常值处理和 Prophet 节假日建模。

经项目脚本裁剪后的 train_subset.csv 实际包含 277,860 条记录，覆盖 5 家门店、33 个品类、1,684 个自然日，日期范围为 2013-01-01 至 2017-08-15。数据总销售量约为 120,686,416.65，日均总销售量约为 71,666.52。累计销售量最高的前五个品类分别为 GROCERY I、BEVERAGES、PRODUCE、CLEANING 和 DAIRY。

### 3.5 可行性分析

从技术可行性来看，Python 生态中已有成熟的数据处理、时间序列分析、机器学习和 Web 开发工具，能够支撑本系统实现。Flask 可用于构建轻量级 Web 应用，Pandas 可用于销售数据处理，statsmodels、Prophet 和 PyTorch 可用于模型构建，ECharts 可用于前端图表展示。

从经济可行性来看，系统主要采用开源工具和本地部署方式，不依赖昂贵商业软件，适合中小商贸企业低成本使用。

从操作可行性来看，系统通过浏览器提供图形化界面，用户无需直接编写代码即可完成数据上传、分析和预测，具有较好的实际应用便利性。

## 第 4 章 系统设计

### 4.1 系统总体架构

本系统采用分层架构设计，自上而下包括用户交互层、业务逻辑层、模型推理层和数据存储层。

用户交互层由 HTML 页面、Bootstrap 样式和 ECharts 图表组成，主要负责文件上传、参数配置、图表展示和结果导出等交互功能。

业务逻辑层由 Flask 路由和 API 接口组成，负责接收前端请求、调用数据处理模块、调用分析模块、启动预测任务、查询任务进度和返回 JSON 数据。

模型推理层包括 ARIMA、Prophet、LSTM 和模型评估模块，负责完成时间序列训练、预测、评估和模型结果组织。

数据存储层采用文件方式存储原始数据、清洗后数据、元数据、模型缓存和预测导出文件。该方式实现简单，适合毕业设计和单机部署场景。


### 4.1.1 项目工程结构与数据流设计

从当前项目实现看，系统并非单一算法脚本，而是围绕销售数据业务流程组织的 Web 应用。项目根目录包含 `app.py`、`config.py`、`modules`、`templates`、`static`、`tests` 和 `data` 等目录。其中 `app.py` 负责 Flask 应用入口、页面路由、API 接口、预测任务调度和导出功能；`config.py` 统一维护数据路径、字段名、模型参数、预测天数、门店与品类范围以及品类中文映射；`modules` 目录按职责拆分为数据处理、销售分析、三类预测模型、模型评估和任务管理模块；`templates` 和 `static` 目录分别承载页面模板、样式、JavaScript 交互和本地化前端依赖；`tests` 目录提供数据处理、模型和 API 的自动化测试。

系统数据流可以概括为七个阶段。第一阶段，用户在首页上传 CSV/XLSX 文件，或直接加载 `data/raw/train_subset.csv` 演示数据。第二阶段，`DataProcessor` 完成字段校验、日期解析、重复记录聚合、负值裁剪、缺失值填充、异常值处理和元数据保存。第三阶段，清洗后的数据被写入 `data/processed`，并通过会话元数据供后续页面读取。第四阶段，分析页调用 `DataAnalyzer` 生成概览指标、趋势序列、月度对比、品类占比、Top 品类、相关性、星期效应和促销效果等结果。第五阶段，预测页根据当前会话数据动态加载品类和门店选项，并对指定序列执行 ADF 平稳性检验。第六阶段，后端创建预测任务，由后台线程训练或加载 ARIMA、Prophet、LSTM 模型，并使用统一评估器计算指标。第七阶段，报告页展示预测曲线、误差对比、雷达图、模型推荐、Prophet 分量和 LSTM 训练损失曲线，同时支持 CSV 导出。

该数据流体现了系统从原始数据到经营辅助信息的转换过程：原始销售记录经过清洗后形成可分析数据，历史数据经过模型训练后形成未来预测结果，模型评估结果再通过图表和表格转换为用户能够理解的决策报告。

### 4.2 功能模块设计

系统主要划分为以下模块。

（1）数据处理模块 DataProcessor。负责读取 CSV/XLSX 数据，完成字段校验、清洗、缺失值处理、异常值处理、特征工程、聚合统计和处理结果保存。

（2）数据分析模块 DataAnalyzer。负责基于清洗后数据生成概览统计、趋势图、月度对比、品类占比、Top 品类排行、相关性热力图、星期效应、促销效果和 STL 季节分解结果。

（3）ARIMA 模型模块 ARIMAModel。负责 ADF 检验、差分阶数判断、ARIMA 参数搜索、模型训练、预测、滚动评估和模型保存加载。

（4）Prophet 模型模块 ProphetModel。负责节假日数据加载、Prophet 训练、未来预测、趋势和季节分量输出、模型保存加载。

（5）LSTM 模型模块 LSTMModel。负责滑动窗口样本构造、网络训练、验证损失记录、递归预测、测试集评估和模型保存加载。

（6）模型评估模块 ModelEvaluator。负责计算 MAE、RMSE、MAPE、SMAPE、R²，生成模型排名、误差对比图和雷达图数据。

（7）任务管理模块 TaskManager。负责维护后台预测任务状态，包括任务创建、进度更新、完成、失败、取消和查询。

（8）前端展示模块。包括 index.html、analysis.html、prediction.html 和 report.html 四个页面，以及 upload.js、analysis.js、prediction.js、report.js 和 common.js 等脚本。

### 4.3 系统流程设计

系统主流程如下。

第一步，用户在首页上传销售数据文件。后端接收文件后调用 DataProcessor 进行加载、校验、清洗和保存，并将清洗数据路径写入会话。

第二步，用户进入分析看板。前端调用分析 API 获取总销量、趋势、月度对比、品类占比和 Top 品类等数据，并使用 ECharts 渲染图表。

第三步，用户进入预测工作台。用户选择品类、门店、预测天数和模型组合。系统可根据所选序列执行 ADF 检验，为 ARIMA 差分阶数提供参考。

第四步，用户启动预测任务。后端创建任务 ID，并在后台线程中执行数据切分、模型训练、模型评估、全量重训、未来预测和结果组织。

第五步，前端轮询任务进度。任务完成后，系统跳转预测报告页，展示预测结果和模型对比。

第六步，用户可在报告页查看推荐模型和预测明细，也可导出 CSV 文件。

### 4.4 数据处理设计

系统要求上传数据至少包含 date、store_nbr、family 和 sales 字段，onpromotion 为可选字段。数据处理流程包括：

（1）文件读取。系统支持 CSV 和 XLSX，不支持旧版 XLS 格式。CSV 读取时依次尝试 UTF-8、UTF-8-SIG 和 GBK 编码。

（2）字段校验。系统检查必要字段是否存在，检查日期字段是否可解析，检查销售字段是否可转换为数值。

（3）重复处理。系统按 date、family、store_nbr 聚合重复记录，对 sales 求和，对 onpromotion 求和。

（4）负值处理。系统将 sales 和 onpromotion 中的负值裁剪为 0，避免负销量对统计分析和模型训练造成影响。

（5）缺失值处理。系统按 family 和 store_nbr 分组处理缺失值。短缺失使用时间插值，中等长度缺失使用前向填充和月度均值补充，长缺失使用中位数填充。

（6）异常值处理。系统结合 7 日滚动中位数和 IQR 规则识别异常高值。若异常发生在节假日或事件日期，则保留该高值，以避免误删真实促销或节假日销售高峰。

### 4.5 接口设计

系统主要 API 如下。

（1）POST /api/upload：上传数据文件，返回 session_id、清洗结果摘要和校验信息。

（2）POST /api/demo/load：加载系统内置的 train_subset.csv 演示数据，便于在答辩或演示环境中无需手动上传即可进入分析和预测流程。

（3）GET /api/data/summary：获取当前会话的数据摘要。

（4）GET /api/data/options：根据当前会话数据返回可选品类和门店列表，前端据此动态生成下拉框，避免固定写死品类或门店。

（5）GET /api/analysis/overview：获取总销量、日均销量、同比增长、最佳品类等概览指标。

（6）GET /api/analysis/trend：按日、周、月粒度获取销售趋势数据。

（7）GET /api/analysis/monthly_comparison：获取月度销售对比数据。

（8）GET /api/analysis/category_pie：获取品类销售占比数据。

（9）GET /api/analysis/top_families：获取销售量排名靠前的品类。

（10）GET /api/analysis/adf：对指定品类和门店序列执行 ADF 检验。

（11）POST /api/predict/start：启动预测任务。

（12）GET /api/predict/progress：查询预测任务进度。

（13）GET /api/predict/result：获取预测结果。

（14）POST /api/predict/cancel：取消预测任务。

（15）GET /api/export/forecast：导出预测结果 CSV。

## 第 5 章 系统实现

### 5.1 开发环境

系统开发环境与开题报告保持一致，主要包括 Python 3.10、Flask、Pandas、NumPy、statsmodels、Prophet、PyTorch、scikit-learn、Bootstrap 和 ECharts。项目在 Windows 环境下开发和测试，前端静态资源 Bootstrap 与 ECharts 存放于 static/vendor 目录，支持离线访问。


### 5.1.1 当前项目实现工作量与模块依据

根据项目文件统计，当前系统包含 Python 代码约 3718 行、HTML 模板约 652 行、前端 JavaScript 约 955 行、CSS 样式约 1873 行，后端页面和 API 路由共 23 个。上述代码规模说明系统实现已经覆盖数据处理、模型预测、Web 交互、可视化展示和自动化测试等多个方面。

后端接口采用统一 JSON 响应格式，成功响应包含 `code`、`message` 和 `data` 字段，错误响应返回具体状态码和提示信息。数据上传接口限制文件类型为 CSV 和 XLSX，拒绝旧版 XLS 文件，并设置 16 MB 文件大小上限。分析接口包括数据摘要、选项获取、概览统计、趋势分析、月度对比、品类占比、Top 品类、相关性、星期规律、季节分解、促销效果和 ADF 检验。预测接口包括任务启动、进度查询、结果获取、任务取消和预测结果导出。前端页面通过 `sessionStorage` 保存会话 ID 和任务 ID，使用户能够在上传、分析、预测和报告页面之间连续操作。

前端实现体现了较完整的业务交互设计。首页支持拖拽上传、文件选择、上传进度展示和演示数据加载；分析页使用并发请求加载概览、选项和图表数据，并支持按时间粒度、品类和门店筛选趋势图；预测页在模型训练前自动执行 ADF 检验，支持选择 ARIMA、Prophet、LSTM 模型组合，并配置部分模型参数；报告页展示预测曲线、模型指标表、误差柱状图、雷达图、推荐模型说明，并在 Prophet 和 LSTM 可用时展示分量曲线和训练损失曲线。这些实现使系统具备从数据导入到结果解释的完整交互闭环。

### 5.2 数据处理模块实现

数据处理模块位于 modules/data_processor.py。DataProcessor 类以文件路径和会话 ID 初始化，通过 load 方法完成数据读取，通过 validate 方法完成字段和内容校验，通过 clean 方法完成清洗主流程。

在清洗过程中，系统首先将日期字段转换为 datetime 类型，将 sales 和 onpromotion 转换为数值类型，然后按照 store_nbr、family、date 排序。对于重复记录，系统使用 groupby 按 date、family、store_nbr 聚合，使同一天同一门店同一品类只保留一条记录。

缺失值处理采用分组策略，避免不同门店或不同品类之间的数据相互污染。对于连续缺失不超过 7 天的情况，使用时间插值；对于连续缺失不超过 30 天的情况，使用前向填充和月度均值补充；对于更长缺失，使用中位数填充。该策略兼顾了短期缺失的连续性和长期缺失的稳健性。

异常值处理结合节假日信息进行判断。系统从 data/raw/holidays_events.csv 中加载 Holiday、Event 和 Additional 类型日期，并过滤 transferred 为 true 的节假日。若某一高销售值出现在节假日或事件日期，则系统倾向于保留该值；否则结合滚动中位数和 IQR 上界识别异常峰值，并用局部中位数替换。

处理完成后，系统将清洗数据、日总销量和元数据写入 data/processed 目录。元数据包含会话 ID、创建时间、清洗文件路径、日总销量文件路径和数据摘要，用于后续分析页面和预测页面读取。

### 5.3 数据分析模块实现

数据分析模块位于 modules/analyzer.py。DataAnalyzer 类接收清洗后的 DataFrame，并围绕销售数据生成多种分析结果。

概览统计包括总销售量、日均销售量、同比增长率、最佳品类、日期范围、门店数量和品类数量。趋势分析接口支持 daily、weekly 和 monthly 三种粒度。当用户选择日粒度但数据跨度较长时，系统会自动降级为周粒度，避免前端图表点数过多导致渲染压力过大。

月度销售对比通过按 year 和 month 分组构建透视表，使用户能够观察不同年份之间的月度变化。品类销售占比通过饼图展示各品类累计销售贡献。Top 品类排行用于突出高贡献品类。相关性热力图通过不同品类的日销售序列相关系数矩阵展示品类之间的联动关系。星期效应分析用于观察不同星期的平均销售差异。促销效果分析则通过促销与非促销样本平均销量对比，辅助判断促销对销量的影响。

### 5.4 ARIMA 模型实现

ARIMA 模型位于 modules/arima_model.py。系统首先使用 ADF 检验判断序列是否平稳。若原序列平稳，则差分阶数 d 取 0；若一阶差分后平稳，则 d 取 1；否则 d 取 2。随后系统在配置的 p_range 和 q_range 中遍历候选参数，使用 AIC 或 BIC 准则选择较优模型。

模型训练完成后，系统可调用 predict 方法生成未来指定天数的预测值、置信区间和模型参数信息。为了避免负销量，预测值和置信区间下界均裁剪为不小于 0。模型评估采用滚动预测方式，每次使用历史序列拟合并预测下一期，再将真实值加入历史序列，尽量模拟实际预测场景。

系统还实现了模型保存和加载功能，通过 joblib 将模型对象序列化到 data/models 目录，为后续缓存复用提供基础。

### 5.5 Prophet 模型实现

Prophet 模型位于 modules/prophet_model.py。系统在初始化时设置趋势灵活度、季节性先验、季节性模式、年季节性、周季节性和是否使用促销回归量等参数。

节假日处理是 Prophet 模块的重要部分。系统读取 holidays_events.csv，将节假日、事件和附加假日转换为 Prophet 所需的 ds、holiday、lower_window 和 upper_window 格式。若节假日文件不存在或加载失败，系统会记录警告并在无节假日特征条件下继续运行，保证系统鲁棒性。

当 use_onpromotion 参数启用且促销序列存在时，系统将 onpromotion 作为外部回归量加入模型。预测时，如果未来促销数据缺失，则默认填充为 0。Prophet 预测结果包括预测值、上下置信区间以及 trend、weekly、yearly、holidays 等分量，便于在报告页展示模型解释信息。

### 5.6 LSTM 模型实现

LSTM 模型位于 modules/lstm_model.py。系统首先根据 seq_len 将时间序列转换为滑动窗口样本，即以前 seq_len 天的销售量预测下一天销售量。训练前使用 MinMaxScaler 基于训练集进行归一化，避免验证集和测试集信息泄漏。

模型网络由 LSTM 层、Dropout 层和全连接层组成。训练过程中使用 MSELoss 作为损失函数，Adam 作为优化器，并使用 ReduceLROnPlateau 调整学习率。EarlyStopping 模块用于记录验证损失较优的模型参数，并在验证损失长期不改善时提前停止训练。

预测时，LSTM 模型采用递归预测方式：先用最近的历史窗口预测下一天，再将预测值加入窗口继续预测后续日期。由于 LSTM 本身不直接输出置信区间，系统报告页仅展示其预测曲线和训练损失曲线。

### 5.7 模型评估与推荐实现

模型评估模块位于 modules/evaluator.py。系统将真实值和预测值转换为 NumPy 数组，并在长度不一致时截断到相同长度，避免指标计算报错。对于 NaN 和无穷值，系统统一转换为安全数值，确保 API 返回 JSON 时不出现 NaN。

在模型对比中，系统计算 MAE、RMSE、MAPE、SMAPE 和 R²，并对 MAE、RMSE、MAPE 进行归一化。综合评分按 MAPE、RMSE、MAE 加权计算，其中 MAPE 权重较高，因为业务人员更容易理解百分比误差。最终系统按综合评分、MAPE、RMSE、MAE 和模型名称排序，选出推荐模型。

### 5.8 后台任务与模型缓存实现

预测任务可能涉及多个模型训练，耗时较长。系统通过 TaskManager 为每次预测创建唯一 task_id，并维护任务状态、进度、当前步骤、模型状态、结果和错误信息。前端每 2 秒调用 /api/predict/progress 查询进度，任务完成后进入报告页获取结果。

为了避免重复训练，系统实现了模型缓存机制。缓存键由模型名称、品类、门店、序列指纹和模型配置共同生成。当相同数据和相同参数再次预测时，系统可从 data/models 目录加载已保存模型和元数据，从而提升响应速度。

### 5.9 前端页面实现

系统前端包括四个主要页面。

首页 index.html 用于数据导入，提供拖拽上传、文件选择、上传进度、数据概览和数据警告展示。同时，为提升答辩演示的稳定性，系统增加“加载内置演示数据”入口，当前环境存在 data/raw/train_subset.csv 时可直接创建演示会话并进入后续分析流程。

分析页 analysis.html 用于销售运营分析，提供时间粒度、品类和门店筛选，并展示 KPI 卡片、趋势图、月度对比图、品类占比图和 Top 品类排行图。品类和门店下拉选项由 /api/data/options 根据当前会话数据动态生成，避免前端固定写死数据范围。

预测页 prediction.html 用于配置预测任务，用户可以选择品类、门店、预测天数、模型组合和部分参数。页面会自动触发 ADF 检验，并在预测任务执行时展示进度条和模型状态。

报告页 report.html 用于展示预测结果，包含预测结果、模型性能和详细分析三个 Tab。用户可以查看未来预测曲线、误差指标、模型对比图、推荐模型说明，并导出 CSV 文件。

## 第 7 章 总结与展望

### 7.1 工作总结

本文围绕“商贸公司销售数据分析与智能预测系统”开展研究与实现，完成了从课题背景分析、技术路线设计、系统架构设计到代码实现和测试验证的完整工作。

在数据处理方面，系统实现了销售数据读取、字段校验、缺失值处理、异常值处理、重复聚合、特征工程和处理结果保存，为后续分析和预测提供了规范数据基础。

在数据分析方面，系统实现了销售概览、趋势分析、月度对比、品类结构、Top 品类排行、相关性分析、星期效应和促销效果分析，使用户能够从多个角度理解销售数据。

在智能预测方面，系统集成 ARIMA、Prophet 和 LSTM 三种模型，分别体现传统统计模型、业务时间序列模型和深度学习模型的特点。系统通过统一评价指标和综合评分对模型进行比较，并在预测报告中给出推荐模型。

在系统实现方面，本文采用 Flask 构建 Web 后端，使用 Bootstrap 和 ECharts 构建前端页面，实现了数据上传、分析看板、预测工作台、预测报告和结果导出等功能，形成了较完整的 Web 应用系统。

在测试验证方面，项目使用 pytest 完成了数据处理、模型模块和 API 接口测试，当前测试结果为 44 个用例全部通过，说明系统核心流程具备较好的可用性和稳定性。

### 7.2 不足之处

由于毕业设计开发周期和实验条件限制，系统仍存在一些不足。

第一，当前系统主要支持单品类、单门店序列预测，虽然适合展示模型流程，但对于企业实际批量预测需求仍需进一步扩展。

第二，LSTM 模型训练耗时较长，且预测效果受参数设置、训练轮数和硬件条件影响较大，当前系统将其作为扩展对比模块，后续仍需进一步调参和优化。

第三，系统采用文件方式保存数据和模型缓存，适合单机演示和小规模使用，但在多用户并发和长期生产环境中，仍需引入数据库、任务队列和权限管理。

第四，论文中部分模型效果需要在统一实验设置下进一步补充，例如选择固定品类和门店，统一预测窗口，记录三种模型在同一测试集上的误差指标和运行耗时。

### 7.3 展望

后续可从以下方向改进系统。

（1）扩展批量预测能力。支持用户一次选择多个品类和多个门店，自动生成批量预测报告。

（2）引入更多外部变量。除节假日和促销外，可加入油价、门店类型、城市、工资日、天气等特征，提高模型对业务场景的表达能力。

（3）扩展模型类型。可引入 XGBoost、LightGBM、随机森林、Temporal Fusion Transformer 等模型，与现有 ARIMA、Prophet、LSTM 形成更丰富的模型库。

（4）优化部署架构。可引入数据库保存用户、任务和预测结果，使用 Celery 或 RQ 管理后台任务，并通过 Docker 完成部署。

（5）完善报表导出。除 CSV 外，可进一步支持 Excel、PDF 和 Word 报告导出，便于企业内部汇报和归档。

（6）增强系统安全性。后续可增加用户登录、权限控制、文件安全扫描和访问日志，提升系统在真实业务环境中的安全性。

综上，本文完成的系统验证了数据分析与智能预测技术在商贸销售场景中的应用价值，为中小商贸企业低成本开展销售数据分析和预测提供了一种可行方案。
